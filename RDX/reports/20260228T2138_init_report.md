## Architecture Observations

1. **Clean core isolation**: `phases/` has zero framework imports. The `manager.ts` uses `EventTarget` (web standard), not React state. This is well-disciplined.

2. **Dual-type PhasePacket**: `title.types.ts` defines a rich `PhasePacket` with selection/alignment/inventory fields. `manager.ts` defines a minimal `PhasePacket` with just from/to/ts/user/identity. These are not the same type — potential confusion point.

3. **Module-level singleton state**: `meta.ts` uses a module-level `state` object (not a class, not a store). This works for single-tab but makes testing harder (no reset between tests without `clearSnapshot()` + re-import).

4. **Exchange event flow is indirect**: Title animation → `onEnterDudael()` → `titleExchange()` → `commitExchange()` (localStorage) → `CustomEvent("dudael:exchange")` → `boot.ts` listener → `initRunMeta()` + `initManager()` + `transition()`. This chain works but is spread across 4 files.

5. **Dev-walk is web-coupled**: `dev-walk.ts` uses `window.addEventListener` directly. If `phases/` is meant to be runtime-agnostic, this should live in `web/` or be guarded.

6. **Path alias `@/workbench/phases/`**: Used in `boot-walk.ts` and `SelectShell.tsx`. Needs a Vite `resolve.alias` or `tsconfig` paths entry in `web/react_app/` to resolve.
