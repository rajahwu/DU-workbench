## Active Suggestions

### 1. Use `readActivePacket()` instead of raw localStorage (High Priority)

Every Shell still manually does `JSON.parse(localStorage.getItem("dudael:active_packet"))`. The barrel already exports `readActivePacket()` from `boot.ts` which handles validation and error cases. After UI FORGE refactor, the pattern now appears in 5 Shell files only (SelectShell, LevelShell, DoorShell, DropShell, DraftLayout) — each a 2-line change. Quick win that eliminates a class of `JSON.parse` errors.

### 2. Make meta.ts testable (High Priority)

The module-level singleton makes unit testing hard. Options:
- Export `resetRunMeta()` for test teardown
- Or factory pattern: `createRunMeta()` returns an instance

This blocks the entire Testing section of the TODO list. With the data layer now providing typed helpers (`getDraftPoolCounts`, `getLevelCombatDeltas`, etc.), these are highly testable — but only if the meta singleton can be reset between test cases.

### 3. Decide title screen strategy (Medium Priority)

Two title implementations exist:
- **`TitleShell/`** (HTML/CSS) — currently routed, handles the exchange flow
- **`TitlePage.tsx`** (PixiJS via `mountTitleScreen`) — orphaned, wraps the procedural animation

The `TitlePage.tsx` has a pre-existing type error (`mountTitleScreen` not exported). Options: (a) Route the PixiJS version and move exchange logic to its `onEnterDudael` callback, (b) Keep HTML/CSS title and remove `TitlePage.tsx`, (c) Use PixiJS as a canvas background behind the HTML title.

### 4. Wire Redux Toolkit for state (Medium Priority)

Redux Toolkit is installed but unused. The current pattern (direct `getRunMeta()` calls + localStorage packet reads) works for prototype but won't scale for:
- Reactive UI updates when alignment changes mid-level
- Cross-component state sharing (e.g., drafted cards visible in Level HUD)
- DevTools time-travel debugging

Now that vessel stats are typed (`VesselPacketStats`) and written to the packet, a Redux `gameSlice` could initialize directly from `packet.player.stats`. The typed data layer makes this straightforward.

### 5. Wire Supabase for profile persistence (Medium Priority)

`loadUserProfile()` in `title-exchange.ts` reads from `localStorage`. The dependency is installed. The exchange seam is clean — just swap the localStorage read for a Supabase query inside `loadUserProfile()`. No other changes needed.

### 6. Expand the card pool (Medium Priority)

`data/cards/pool.ts` has 10 prototype cards (5 Light, 5 Dark). The `DraftCard` schema supports rich features that aren't yet exercised:
- **Vessel interactions** — Only `LANTERN_AT_THE_THRESHOLD` and `ROOT_WHISPER` have them
- **Visibility manifests** — All prototype cards use the same default manifest; design cards with unique insight gating
- **Keeper signals** — Only the 2 rich cards have `keeperSignal`; these should influence DraftApproach dialogue
- **Reckoning comments** — Same; these should replace the hardcoded `keeperComment` logic in DraftReckoning
- **Depth ranges** — Cards should appear/disappear based on depth
- **Rarity tiers** — Uncommon/rare/relic cards for deeper runs

Target: 20+ cards with at least 6 carrying vessel interactions and keeper signals.

### 9. Add proper types for cartridge interfaces (Low Priority)

The `LevelShell` is a tap-grid prototype. As more cartridge types are added (top-down, platformer, puzzle per the staging phase diagram), define a `CartridgeInterface`:

```ts
export type CartridgeResult = { survived: boolean; points: number; alignment?: Partial<Alignment> };
export type CartridgeProps = { depth: number; vesselId: VesselId; combatDeltas: CombatDeltas; draftedCards: DraftCard[]; onComplete: (result: CartridgeResult) => void };
```

Now that `getLevelCombatDeltas()` provides typed combat config, the cartridge interface can accept it as a prop rather than each cartridge computing its own.

### 10. Extract lore content to data layer (Low Priority)

Theology text, keeper dialogue lines, vessel approach notes, and reckoning comments are still hardcoded across shells. A `data/lore/` directory could hold:
- Vessel theology (SelectShell)
- Keeper approach lines (DraftApproach)
- Keeper reckoning comments (DraftReckoning — partially in card schema, partially hardcoded)
- Level names and flavor text (LevelShell)
- Drop outcome text (DropShell)

This would make lore content editable by a writer without touching React components.
